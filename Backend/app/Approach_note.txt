# Book Management System – Validation Enhancement Approach Note

## Objective

The Book Management System allowed users to create books with invalid values for the following fields:

1. **Date Field**

   * Invalid date formats could be entered.
   * Future dates could be entered.

2. **Cost Field**

   * Zero or negative values could be entered.

The objective was to enforce proper validation rules to improve data integrity and prevent invalid records from being stored in the database.

---

# Issue 1: Date Validation

## Problem Analysis

Initially, the `date` field was stored without proper validation.

Examples of invalid records:

```json
{
  "publisher": "ABC",
  "name": "Python Basics",
  "date": "18-23-120",
  "Cost": 200
}
```

Problems:

* Month value `23` is invalid.
* Year `120` is not meaningful.
* Future dates such as `03-02-2045` were also accepted.

This happened because the application was treating the date as a normal string and no business validation was applied.

---

## Solution

### Step 1: Use Python Date Type

The schema was modified to use Python's `date` type instead of a plain string.

```python
from datetime import date

class BookBase(BaseModel):
    date: date
```

### Benefit

Pydantic automatically validates date values.

Examples:

Valid:

```json
{
  "date": "2024-04-02"
}
```

Invalid:

```json
{
  "date": "18-23-120"
}
```

Response:

```json
{
  "error": "Invalid date format"
}
```

This prevents malformed dates from reaching the database.

---

### Step 2: Prevent Future Dates

A custom validator was implemented.

```python
@field_validator("date")
@classmethod
def validate_date(cls, value):
    if value > date.today():
        raise ValueError(
            "Future dates are not allowed"
        )
    return value
```

### Benefit

Books cannot be created with dates that have not yet occurred.

Example:

```json
{
  "date": "2045-02-03"
}
```

Response:

```json
{
  "error": "Future dates are not allowed"
}
```

# Issue 2: Cost Validation

## Problem Analysis

Initially, users could submit values such as:

```json
{
  "Cost": 0
}
```

or

```json
{
  "Cost": -100
}
```

These values are not logically valid because a book cost should always be greater than zero.

---

## Solution

Pydantic's `Field()` validation was used.

```python
from pydantic import Field

Cost: float = Field(gt=0)
```

### Explanation

`gt=0`

means:

```text
Cost > 0
```

Only positive values are accepted.

---

## Validation Results

Valid:

```json
{
  "Cost": 100
}
```

Accepted.

---

Invalid:

```json
{
  "Cost": 0
}
```

Rejected.

---

Invalid:

```json
{
  "Cost": -50
}
```

Rejected.

---

# Final Schema

```python
from pydantic import BaseModel, Field, field_validator
from datetime import date

class BookBase(BaseModel):
    publisher: str
    name: str
    date: date
    Cost: float = Field(gt=0)

    @field_validator("date")
    @classmethod
    def validate_date(cls, value):
        if value > date.today():
            raise ValueError(
                "Future dates are not allowed"
            )
        return value
```

---

# Benefits Achieved

1. Improved data quality.
2. Prevented invalid date formats.
3. Prevented future publication dates.
4. Prevented zero and negative book costs.
5. Reduced chances of storing incorrect data in the database.
6. Leveraged Pydantic's built-in validation features for cleaner and maintainable code.

---

# Conclusion

The Book Management System was enhanced by introducing schema-level validation using Pydantic. Date validation ensures that only valid and realistic dates are accepted, while cost validation guarantees that book prices remain positive. These changes improve application reliability, maintain data consistency, and enforce business rules before data reaches the database.
